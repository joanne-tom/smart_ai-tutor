As our major project, we are intending to build Smart AI Tutor. Smart AI Tutor is a modular AI-powered classroom assistant built on five structured layers: a Knowledge Layer (RAG), an MCP-based Flask backend orchestration layer, a Session & Engagement tracking layer, a Flutter-based Application layer (Faculty + Student interfaces), and a Reinforcement Learning layer added after core stabilization. At the beginning of the semester, faculty upload syllabus PDFs and personal notes, which are extracted, cleaned, chunked, embedded, and stored in a vector database with detected topic headings to enable dropdown topic selection later. When a faculty member is suddenly absent, they log in, select subject, date, and topic(s), and schedule a session; the backend creates a session entry and notifies students. Students log in using name and roll number, view the scheduled session, and enter a chat-style interface where the system retrieves relevant syllabus content through RAG, generates grounded explanations, displays them on the laptop/projector, and plays audio using text-to-speech. When a student interrupts with a doubt, the backend embeds the query, retrieves top relevant chunks from the vector store, generates a strictly syllabus-grounded answer, and continues the lecture. The MCP layer, implemented via structured Flask APIs (login, upload_notes, get_topics, schedule_session, start_session, ask_question, mark_attendance, log_engagement), ensures secure tool access, session synchronization, attendance tracking, and clean separation between AI logic and data control. Engagement tracking monitors interruptions, quiz accuracy, idle time, background app events, and attendance duration to maintain an engagement score (e.g., positive for correct responses, negative for repeated doubts or app exits). After MCP is stable, a lightweight Reinforcement Learning module (preferably contextual bandits) is integrated, where the state includes engagement score, interruption count, quiz accuracy, and topic difficulty, actions include detailed explanation, example-based explanation, simplification, summary, or verification question, and rewards are based on learning improvement and engagement; RL does not modify content but selects teaching strategy, while RAG ensures factual correctness. Development should proceed in phases: first stabilize MCP + RAG + basic Flutter interfaces (faculty scheduling, student login, chat, audio playback), then integrate engagement tracking and RL adaptation, followed by synchronization polish and demo stabilization. With three team members dividing backend, RAG/database, and Flutter responsibilities, achieving an 80–85% functional academic demo by mid-March is realistic, provided scope is frozen, UI remains simple, and production-level features are avoided.
